## 전제 환경

EC2 인스턴스:

| 경로 | 용도 |
|------|------|
| `/opt/pangchelin/` | 프로젝트 루트 (git checkout) |
| `/opt/pangchelin/.env.production` | 운영 환경변수 (660 권한, ec2-user:docker) |
| `/opt/pangchelin/docker-compose.prod.yml` | 운영 컴포즈 |

쉘에서 자주 export 해두면 편리:

```bash
export AWS_REGION=ap-northeast-2
export ECR_REGISTRY=<account-id>.dkr.ecr.ap-northeast-2.amazonaws.com
export PROJECT_DIR=/opt/pangchelin
```

EC2 접속:

```bash
# 권장: SSM Session Manager (SSH 포트 22 안 열어도 됨)
aws ssm start-session --target <instance-id> --region $AWS_REGION

# 또는 SSH (보안그룹에 22 열려 있을 때)
ssh -i ~/.ssh/pangchelin.pem ec2-user@<public-ip>
```

## 일상 운영

### 컨테이너 상태 확인

```bash
cd $PROJECT_DIR
docker compose -f docker-compose.prod.yml ps
```

각 컨테이너의 STATUS 가 `Up X minutes (healthy)` 여야 정상.

### 로그 보기

```bash
# 실시간 (Ctrl+C 로 종료)
docker compose -f docker-compose.prod.yml logs -f api
docker compose -f docker-compose.prod.yml logs -f web
docker compose -f docker-compose.prod.yml logs -f nginx

# 최근 100줄만
docker compose -f docker-compose.prod.yml logs --tail 100 api

# 여러 서비스 동시
docker compose -f docker-compose.prod.yml logs -f api web nginx

# CloudWatch 로 보낸 경우 (awslogs driver 활성화 시)
aws logs tail /ec2/pangchelin --follow --region $AWS_REGION
```

### 재시작

```bash
# 단일 서비스
docker compose -f docker-compose.prod.yml restart api

# 전체
docker compose -f docker-compose.prod.yml restart

# 컨테이너 안 들어가서 디버깅
docker compose -f docker-compose.prod.yml exec api sh
```

### 디스크 / 메모리 상태

```bash
# EC2 자원 사용량
df -h                                    # 디스크
free -m                                  # 메모리
docker stats --no-stream                 # 컨테이너별 CPU/메모리

# Docker 사용량 정리 (안 쓰는 이미지/볼륨)
docker system df
docker system prune -f                   # 안 쓰는 컨테이너/네트워크 정리
docker image prune -af                   # 안 쓰는 이미지 정리 (주의: 롤백용 이전 SHA 도 사라짐)
```

## 배포

### 사전 준비 (한 번만)

1. ECR 리포지토리 2개 생성: `pangchelin-api`, `pangchelin-web`
2. EC2 의 IAM 인스턴스 프로파일에 `AmazonEC2ContainerRegistryReadOnly` 정책 부여
3. EC2 안에 `aws-cli`, `docker`, `docker compose plugin` 설치

### 배포 절차 (수동, 매 배포마다)

배포할 git SHA 결정. 짧은 SHA 도 OK (예: `a3f8b21`).

```bash
# 0) 로컬에서 ECR 에 이미지 푸시 (수동)
#    학생 프로젝트 단계에선 로컬 빌드 + 푸시도 충분
GIT_SHA=$(git rev-parse --short HEAD)

aws ecr get-login-password --region $AWS_REGION \
  | docker login --username AWS --password-stdin $ECR_REGISTRY

docker build -f apps/api/Dockerfile \
  -t $ECR_REGISTRY/pangchelin-api:$GIT_SHA \
  -t $ECR_REGISTRY/pangchelin-api:latest .

docker build -f apps/web/Dockerfile \
  --build-arg NEXT_PUBLIC_API_URL=https://<도메인>/api \
  --build-arg NEXT_PUBLIC_KAKAO_MAP_KEY=<카카오 JS 키> \
  -t $ECR_REGISTRY/pangchelin-web:$GIT_SHA \
  -t $ECR_REGISTRY/pangchelin-web:latest .

docker push $ECR_REGISTRY/pangchelin-api:$GIT_SHA
docker push $ECR_REGISTRY/pangchelin-api:latest
docker push $ECR_REGISTRY/pangchelin-web:$GIT_SHA
docker push $ECR_REGISTRY/pangchelin-web:latest

# 1) EC2 SSH/SSM 접속

# 2) (스키마 변경 있는 경우) 마이그레이션
cd $PROJECT_DIR
./scripts/migrate-prod.sh deploy

# 3) 배포
./scripts/deploy.sh $GIT_SHA

# 4) 외부에서 접속 확인
curl -I https://<도메인>/
curl -s https://<도메인>/api/health
```

### 마이그레이션이 필요한지 확인

```bash
./scripts/migrate-prod.sh status
```

`Database schema is up to date!` 메시지면 마이그레이션 불필요.

### 배포 직전에 RDS 스냅샷 (위험한 변경 시)

```bash
./scripts/backup-rds.sh "before-$GIT_SHA"
# 5~30분 기다린 후 배포 진행
```

## 롤백

```bash
# 0) ECR 에서 직전 안정 SHA 확인
aws ecr describe-images --repository-name pangchelin-api --region $AWS_REGION \
  --query 'sort_by(imageDetails,& imagePushedAt)[-10:].imageTags' --output table

# 1) 롤백
./scripts/rollback.sh <previous-sha>
```

⚠ **주의**: DB 마이그레이션은 자동으로 되돌리지 않음. 새 배포에서 스키마 변경이 있었다면:
- 새 스키마와 호환되는 이전 SHA 인지 확인
- 호환 안 되면 마이그레이션 down 필요 (Prisma 는 수동 SQL 작성)
- 가능하면 RDS 스냅샷에서 PITR 로 시점 복원이 더 안전

## DB (RDS)

### 외부에서 직접 접속 (디버깅)

EC2 의 보안그룹만 RDS 5432 허용 → 로컬에서 접속하려면 EC2 통한 SSH 터널:

```bash
ssh -i ~/.ssh/pangchelin.pem -L 5432:<rds-endpoint>:5432 ec2-user@<ec2-public-ip>

# 다른 터미널
psql "postgresql://<user>:<pw>@localhost:5432/pangchelin"
```

또는 EC2 안에서 직접:

```bash
# postgres-client 가 EC2 에 깔려 있어야 함
psql "$DATABASE_URL"
```

### 마이그레이션

```bash
./scripts/migrate-prod.sh status     # 현황
./scripts/migrate-prod.sh deploy     # 적용
```

수동으로 실행하려면:

```bash
docker compose -f docker-compose.prod.yml --env-file .env.production run --rm \
  api ./node_modules/.bin/prisma migrate deploy --schema=prisma/schema.prisma
```

### 시드 데이터 (운영에서는 1회만)

운영 이미지에는 `ts-node` 가 없어서 `prisma:seed` 를 그대로 못 돌림. 옵션:

1. **dev 이미지로 1회 seed** (간단):
   ```bash
   docker run --rm --env-file .env.production \
     -v $PROJECT_DIR:/workspace -w /workspace/apps/api \
     node:20-alpine sh -c "corepack enable && pnpm install --frozen-lockfile && pnpm prisma:seed"
   ```
2. **prisma db execute 로 SQL 직접 실행** (안정적):
   ```bash
   docker compose -f docker-compose.prod.yml run --rm api \
     ./node_modules/.bin/prisma db execute --file ./prisma/seed.sql --schema=prisma/schema.prisma
   ```

## 백업 / 복원

### 자동 백업 (RDS 기본)

- RDS 콘솔 → DB instance → Maintenance & backups 에서 보관 기간 조정 (기본 7일, 권장 14~35일)
- Point-in-Time Recovery (PITR) 가능: 백업 보관 기간 내 임의 시점으로 복원

### 수동 스냅샷

```bash
./scripts/backup-rds.sh "before-major-migration"
```

수동 스냅샷은 **삭제 전까지 영구 보관** → 비용 누적. 1년 이상 묵힌 스냅샷은 정기적으로 정리:

```bash
aws rds describe-db-snapshots --snapshot-type manual --region $AWS_REGION \
  --query 'sort_by(DBSnapshots,& SnapshotCreateTime)[*].[DBSnapshotIdentifier,SnapshotCreateTime,AllocatedStorage]' \
  --output table
```

### 복원 (RDS 콘솔에서 진행 권장)

콘솔이 가장 안전. 절차:

1. RDS 콘솔 → Snapshots → 복원할 스냅샷 선택 → "Restore snapshot"
2. **새 인스턴스로 복원됨** (기존 인스턴스 덮어쓰기 X)
3. 새 엔드포인트 확인 → `.env.production` 의 `DATABASE_URL` 갱신
4. `./scripts/deploy.sh` 로 재시작 (env 반영 위해)
5. 검증 후 옛 인스턴스 삭제

### S3 (이미지)

S3 버킷에 **버저닝 활성화** 필수 — 잘못 덮어쓴 파일도 복원 가능. 콘솔에서:
- Properties → Bucket Versioning → Enable
- Lifecycle rule → 30일 지난 비현재 버전 삭제 (스토리지 절감)

## 장애 대응

### 알람 트리거 시 1차 체크

| 알람 | 확인 |
|------|------|
| EC2 CPU > 70% 5분 | `docker stats` 로 어느 컨테이너인지 확인 → api 라면 `top` 으로 N+1 쿼리 의심 |
| EC2 메모리 > 85% | `free -m`, `docker stats` → `t3.medium` 한계라면 인스턴스 업그레이드 검토 |
| EC2 CPU credit < 100 | 버스트 한계 임박 → 트래픽 지속되면 `t3.medium` → `m5.large` 로 변경 |
| RDS CPU > 70% | RDS Performance Insights → Top SQL 확인. 인덱스 부족 의심 |
| RDS FreeableMemory < 200MB | `db.t4g.small` → `db.t4g.medium` 업그레이드 |
| RDS Connections > 60 | 앱 인스턴스 늘었는지 / connection leak 확인 |

### 자주 발생 시나리오

#### 1. API 가 5xx 응답

```bash
docker compose -f docker-compose.prod.yml logs --tail 100 api
```

흔한 원인:
- DB 연결 끊김 → `DATABASE_URL` 확인, RDS 보안그룹 / 가용성 확인
- 메모리 부족 → 컨테이너 재시작 (`docker compose restart api`), 근본은 인스턴스 업그레이드
- 앱 코드 에러 → 로그의 stacktrace 확인 → 핫픽스 PR / 이전 SHA 로 롤백

#### 2. Web (Next.js) 가 안 뜸

```bash
docker compose -f docker-compose.prod.yml logs --tail 50 web
```

- `NEXT_PUBLIC_*` 가 잘못 박힘 → 새 build-arg 로 이미지 재빌드 + 배포
- standalone 누락 → Dockerfile 확인 (출력 trace root)

#### 3. Nginx 502 Bad Gateway

```bash
docker compose -f docker-compose.prod.yml logs nginx
```

- upstream (`web:3000` / `api:4000`) 컨테이너가 down/unhealthy
- 같은 docker network 안에 있는지 확인 (`docker network inspect pangchelin-prod`)

#### 4. 디스크 가득 참

```bash
df -h
docker system df
docker system prune -af --volumes        # 주의: 안 쓰는 볼륨까지 삭제
journalctl --vacuum-size=500M            # 시스템 로그 정리
```

#### 5. 외부에서 접속 안 됨

체크 순서:
1. EC2 인스턴스 상태 (`aws ec2 describe-instance-status`)
2. EC2 보안그룹 80/443 인바운드
3. CloudFront 배포 상태 (Distribution status: Deployed)
4. Route 53 ALIAS 레코드가 CloudFront 가리키는지

## SSL 인증서

### CloudFront + ACM (기본 권장)

- ACM 인증서는 **us-east-1 리전**에서 발급 (CloudFront 요구사항)
- ACM 자동 갱신 (DNS 검증이라면 영구 자동 갱신, 별도 작업 불필요)
- 갱신 60일 전 이메일 알림 옴 (Route 53 사용 시 자동)

확인:

```bash
aws acm list-certificates --region us-east-1
aws acm describe-certificate --certificate-arn <arn> --region us-east-1 \
  --query 'Certificate.[NotAfter,Status,RenewalEligibility]' --output table
```

### 직접 SSL (Let's Encrypt)

CloudFront 안 쓰고 EC2 nginx 에서 직접 SSL 종단할 경우:

```bash
# certbot 컨테이너 1회 발급
docker run --rm -v $PROJECT_DIR/docker/nginx/certbot/conf:/etc/letsencrypt \
  -v $PROJECT_DIR/docker/nginx/certbot/www:/var/www/certbot \
  certbot/certbot certonly --webroot -w /var/www/certbot -d <도메인> \
  --email <이메일> --agree-tos --no-eff-email

# default.conf 의 443 server 블록 주석 해제
docker compose -f docker-compose.prod.yml exec nginx nginx -s reload

# 갱신 cron (매일 새벽 3시)
echo "0 3 * * * docker run --rm -v $PROJECT_DIR/docker/nginx/certbot/conf:/etc/letsencrypt -v $PROJECT_DIR/docker/nginx/certbot/www:/var/www/certbot certbot/certbot renew --quiet && docker compose -f $PROJECT_DIR/docker-compose.prod.yml exec nginx nginx -s reload" | crontab -
```

## 자주 쓰는 명령어 모음

```bash
# 전체 상태 한 번에 확인
docker compose -f docker-compose.prod.yml ps && \
  docker stats --no-stream && \
  curl -s http://localhost/api/health

# 컨테이너 재기동 (한 줄)
docker compose -f docker-compose.prod.yml restart api && \
  sleep 5 && \
  curl -s http://localhost/api/health

# 새 SHA 배포
./scripts/deploy.sh <git-sha>

# 마이그레이션
./scripts/migrate-prod.sh deploy

# 롤백
./scripts/rollback.sh <previous-sha>

# 수동 백업
./scripts/backup-rds.sh "before-feature-x"

# 컨테이너 내부 디버깅
docker compose -f docker-compose.prod.yml exec api sh
docker compose -f docker-compose.prod.yml exec web sh

# RDS 직접 쿼리 (EC2 안에서)
docker compose -f docker-compose.prod.yml run --rm api \
  ./node_modules/.bin/prisma db execute --stdin --schema=prisma/schema.prisma <<< 'SELECT COUNT(*) FROM "Restaurant";'
```
